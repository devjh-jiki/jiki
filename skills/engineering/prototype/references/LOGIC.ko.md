# 로직 프로토타입 (Logic Prototype)

사용자가 손으로 상태 모델을 굴려 볼 수 있는 작은 인터랙티브 터미널 앱. 질문이 **비즈니스 로직, 상태 전이, 또는 데이터 형태**에 관한 것일 때 사용한다 — 종이 위에선 합리적으로 보이지만 실제 케이스로 밀어붙여야만 잘못된 느낌이 드는 종류.

## 이 형태가 맞을 때

- "X 다음 Y 인 엣지 케이스를 이 상태 기계가 처리하는지 모르겠다."
- "이 데이터 모델이 실제로 ... 케이스를 표현하게 해주나?"
- "쓰기 전에 API 가 어떻게 생겨야 할지 가늠해 보고 싶다."
- 사용자가 **버튼을 누르고 상태가 바뀌는 걸 지켜보고** 싶은 모든 것.

질문이 "이게 어떻게 생겨야 하나" 면 — 틀린 가지. [UI.md](UI.ko.md) 를 사용한다.

## 과정

### 1. 질문을 명시한다

코드를 쓰기 전에, 어떤 상태 모델과 어떤 질문을 프로토타이핑하는지 적는다. 한 문단, 프로토타입의 README 나 파일 맨 위 주석에. 틀린 질문에 답하는 로직 프로토타입은 순전한 낭비다 — 나중에 확인할 수 있게 질문을 명시한다, 사용자가 지금 보고 있든 자리를 비우고 돌아오든.

### 2. 언어를 고른다

호스트 프로젝트가 쓰는 것을 쓴다. 프로젝트에 명확한 런타임이 없으면(예: 문서 레포) 묻는다.

도구는 프로젝트의 기존 관례에 맞춘다 — 프로토타입만을 위해 새 패키지 매니저나 런타임을 추가하지 않는다.

### 3. 로직을 이식 가능한 모듈로 격리한다

실제 로직 — 질문에 답하는 부분 — 을 나중에 들어내 실제 코드베이스에 떨굴 수 있는 작고 순수한 인터페이스 뒤에 둔다. 그 주변의 TUI 는 버릴 것이지만; 로직 모듈은 아니어야 한다.

맞는 형태는 질문에 달렸다:

- **순수 리듀서** — `(state, action) => state`. 액션이 이산 이벤트이고 상태가 단일 값일 때 좋다.
- **상태 기계** — 명시적 상태와 전이. "지금 어떤 액션이 합법인가" 가 질문의 일부일 때 좋다.
- **평범한 데이터 타입에 대한 작은 순수 함수 집합.** 암묵적 현재 상태가 없을 때 좋다 — 그냥 변환들.
- **명확한 메서드 표면을 가진 클래스나 모듈** — 로직이 진짜로 지속되는 내부 상태를 소유할 때.

TUI 에 엮기 가장 쉬운 형태가 *아니라* 묻는 질문에 가장 맞는 형태를 고른다. 순수하게 유지한다: I/O 없음, 터미널 코드 없음, 제어 흐름을 위한 `console.log` 없음. TUI 가 그것을 import 하고 호출한다; 반대 방향으로는 아무것도 흐르지 않는다.

이것이 프로토타입을 자기 수명 너머로 유용하게 만든다. 질문이 답해지면, 검증된 리듀서 / 기계 / 함수 집합을 실제 모듈로 들어 올릴 수 있다 — TUI 껍데기는 지워진다.

### 4. 상태를 드러내는 가장 작은 TUI 를 만든다

**가벼운 TUI** 로 만든다 — 매 틱마다 화면을 지우고(`console.clear()` / `print("\033[2J\033[H")` / 동등물) 프레임 전체를 다시 렌더한다. 사용자는 항상 끝없이 늘어나는 스크롤백이 아니라 하나의 안정된 화면을 봐야 한다.

각 프레임은 이 순서로 두 부분을 가진다:

1. **현재 상태**, 보기 좋고 diff 하기 좋게 출력(한 줄에 한 필드, 또는 포맷된 JSON). 필드 이름이나 섹션 헤더는 **굵게**, 덜 중요한 맥락(타임스탬프, ID, 파생 값)은 **흐리게**. 네이티브 ANSI 이스케이프 코드면 충분하다 — `\x1b[1m` 굵게, `\x1b[2m` 흐리게, `\x1b[0m` 리셋. 프로젝트에 이미 있지 않은 한 스타일링 라이브러리를 끌어올 필요 없다.
2. **키보드 단축키**, 맨 아래 나열: `[a] add user  [d] delete user  [t] tick clock  [q] quit`. 키를 굵게 설명을 흐리게, 또는 반대로 — 깔끔하게 읽히는 쪽으로.

동작:

1. **상태 초기화** — 메모리 내 단일 객체/구조체. 시작 시 첫 프레임을 렌더한다.
2. **한 번에 한 키스트로크(또는 한 줄)를 읽고**, 상태를 변형하는 핸들러로 디스패치한다.
3. 매 액션 후 프레임 전체를 **다시 렌더**한다 — 덧붙이지 말고 교체한다.
4. **종료까지 반복.**

프레임 전체가 한 화면에 맞아야 한다.

### 5. 한 명령으로 실행 가능하게

프로젝트의 기존 태스크 러너(`package.json` 스크립트, `Makefile`, `justfile`, `pyproject.toml`)에 스크립트를 추가한다. 사용자가 `pnpm run <prototype-name>` 또는 동등물을 실행하게 — 경로를 외울 필요가 절대 없게.

호스트 프로젝트에 태스크 러너가 없으면, 그냥 명령을 프로토타입 README 맨 위에 둔다.

### 6. 넘긴다

사용자에게 실행 명령을 준다. 그들이 직접 굴린다; 흥미로운 순간은 그들이 "잠깐, 이게 가능하면 안 되는데" 또는 "어, X 가 다를 줄 알았는데" 라고 말할 때다 — 그게 _아이디어_의 버그이고, 그게 전체 핵심이다. 새 액션을 추가하고 싶어 하면 추가한다. 프로토타입은 진화한다.

### 7. 답을 잡는다

프로토타입이 제 일을 했으면, 질문에 대한 답만이 간직할 가치가 있다. 사용자가 곁에 있으면 무엇을 배웠는지 묻는다. 아니면 프로토타입 옆에 `NOTES.md` 를 남겨 프로토타입이 지워지기 전에 답을 채울 수 있게(또는 세션을 지켜봤다면 당신이 채우게) 한다.

## 안티패턴

- **테스트를 추가하지 마라.** 테스트가 필요한 프로토타입은 더는 프로토타입이 아니다.
- **실제 데이터베이스에 엮지 마라.** 질문이 구체적으로 영속성에 관한 게 아니면 메모리 내 저장소를 쓴다.
- **일반화하지 마라.** "나중에 X 를 지원하고 싶으면" 없음. 프로토타입은 한 질문에 답한다.
- **로직과 TUI 를 뭉개지 마라.** 리듀서 / 상태 기계가 `console.log`, 프롬프트, 또는 터미널 이스케이프 코드를 참조하면 더는 이식 가능하지 않다. TUI 를 순수 모듈 위의 얇은 껍데기로 유지한다.
- **TUI 껍데기를 프로덕션에 보내지 마라.** 껍데기는 터미널에서 손으로 굴리도록 최적화됐다. 그 뒤의 로직 모듈이 간직할 가치가 있는 부분이다.
