# 테스트 주도 개발 (TDD)

동작을 테스트 먼저, 수직 슬라이스 단위로 만든다. 좋은 테스트는 오래가는 자산이고, 나쁜 테스트는 모든 향후 변경을 느리게 하는 부채다. 개발자이자 오너로서 테스트 스위트를 형식 채우기가 아니라 리스크 관리로 다룬다.

## 철학

**핵심 원칙**: 테스트는 구현 디테일이 아니라 공개 인터페이스를 통해 동작을 검증한다. 구현은 완전히 바뀌어도 테스트는 그대로여야 한다.

- **좋은 테스트**는 통합 스타일이다: 공개 API 로 실제 코드 경로를 실행하고 명세처럼 읽힌다("유효한 장바구니로 결제할 수 있다"). 리팩터에 살아남는다.
- **나쁜 테스트**는 구현에 결합한다: 내부 협력자를 목킹하거나 내부 구조를 검증한다. 경고 신호: 동작은 안 바뀌었는데 리팩터하면 테스트가 깨진다.

## 안티패턴: 수평 슬라이스

**모든 테스트를 먼저 쓰고 그다음 모든 구현을 쓰지 않는다.** 한꺼번에 쓴 테스트는 *상상한* 동작을 검증하고, 사물의 *형태*를 테스트하며, 실제 변경에 둔감해진다.

```
틀림(수평):  RED: test1..test5  그다음  GREEN: impl1..impl5
맞음(수직):  RED→GREEN test1→impl1, 그다음 test2→impl2, ...
```

각 테스트는 직전 사이클에서 배운 것에 반응한다.

## 흐름

### 1. 계획
- 공개 인터페이스와 가장 중요한 동작을 확인한다. **모든 걸 테스트할 수 없다** — 핵심 경로와 복잡한 로직에 집중한다.
- 깊은 모듈(작은 인터페이스, 깊은 구현)을 선호해 테스트가 하나의 깔끔한 seam 을 지나게 한다. (`codebase-design` 스킬 참고.)
- 테스트할 동작을 나열한다(구현 단계가 아니라). 사용자 승인을 받는다.

### 2. 트레이서 불릿
첫 동작에 대해 테스트 하나 작성 → 실패(RED) → 최소 코드 작성 → 통과(GREEN). 경로가 엔드투엔드로 동작함을 증명한다. 테스트의 *입력 데이터*는 다음 구현 조각이 반드시 필요해지도록 고른다 — 예: `333 * 0.95 = 316.35`을 일부러 골라 반올림 코드가 필요하게 만든다. 지금 코드를 실패시킬 수 없는 입력의 테스트는 아무것도 끌어내지 못한다.

### 3. 점진 루프
남은 각 동작마다: RED → GREEN. 한 번에 하나, 통과할 만큼의 코드만, 투기적 기능 없이. 가끔 새 테스트가 앞선 코드로 이미 커버돼 바로 통과하기도 한다 — 괜찮다. 그건 회귀 잠금이지 프로세스 실패가 아니다. 그대로 두고 넘어간다.

### 4. 리팩터 (GREEN 일 때만)
중복 추출, 모듈 심화, 자연스러운 곳에 SOLID 적용, 각 단계 후 테스트 재실행. **RED 일 때 절대 리팩터하지 않는다.** 코드가 이미 깨끗하면 리팩터 단계는 건너뛴다 — 모든 사이클에 리팩터가 필요한 건 아니다.

## 사이클별 체크리스트

```
[ ] 테스트가 구현이 아니라 동작을 기술한다
[ ] 테스트가 공개 인터페이스만 사용한다
[ ] 테스트가 내부 리팩터에 살아남는다
[ ] 이 테스트에 필요한 최소 코드다
[ ] 투기적 기능을 추가하지 않았다
```

## 오너 / 리더십 렌즈

정확성을 넘어 테스트를 비즈니스 결정으로 저울질한다:

- **어디에 투자할까**: 결함 비용이 가장 큰 곳(결제, 인증, 데이터 무결성, 고객 대면, 되돌릴 수 없는 것)에 테스트 노력을 둔다. 고치기 싸고 트래픽 적은 경로의 전수 커버리지는 건너뛴다.
- **비가역 부작용은 우선순위가 아니라 설계 신호다.** 카드 결제, 이메일 발송, 데이터 삭제 — 이런 건 테스트에서 실행하면 안 된다. 여기서 high-priority란 "먼저 testable하게 만들기"를 뜻한다: 게이트웨이/발송기를 의존성으로 주입하고(`codebase-design` 참고) 그 주변 가드 로직을 테스트한다(예: 잘못된 입력에서 게이트웨이가 호출되지 않음을 단언). 부작용 자체는 통합 테스트의 몫이다.
- **문서로서의 테스트**: 동작 수준 스위트는 온보딩 자료이자 미래 유지보수자(미래의 당신과 에이전트 포함)와의 계약이다.
- **속도 vs 안전**: 모든 리팩터마다 깨지는 스위트는 팀을 *느리게* 한다. 테스트가 짐이면 잘못된 레이어를 테스트하는 것이다; 증상이 아니라 seam 을 고친다.
- **커버리지 %가 아니라 잡아낸 회귀를 측정한다.** 구현 디테일의 높은 커버리지는 거짓 자신감이다.
