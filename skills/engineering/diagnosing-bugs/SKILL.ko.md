# 버그 진단

어려운 버그를 위한 규율. 명시적으로 정당화될 때만 단계를 건너뛴다. 가장 큰 실수는 테스트할 방법이 없는 채로 가설로 건너뛰는 것이다.

## Phase 1 — 피드백 루프 만들기

**이것이 스킬의 핵심이다.** *이* 버그에서 red 가 되는 단단한 pass/fail 신호가 있으면 원인을 찾는다. 없으면 코드를 아무리 노려봐도 소용없다. 여기서 공격적이고 창의적으로, 포기하지 않는다.

만드는 방법, 대략 선호 순:

1. 버그에 닿는 seam 에서의 **실패 테스트**.
2. 실행 중인 dev 서버에 대한 **curl / HTTP 스크립트**.
3. 픽스처로 **CLI 호출**, stdout 을 known-good 과 diff.
4. DOM/콘솔/네트워크를 검증하는 **헤드리스 브라우저 스크립트**.
5. 저장한 요청/페이로드/이벤트 로그를 코드 경로에 격리 **재생**.
6. 한 함수 호출로 버그 경로를 실행하는 **일회용 하네스**.
7. "가끔 틀린 출력"에는 **프로퍼티/퍼즈 루프**.
8. 두 known 상태 사이에서 생겼으면 **이분 탐색 하네스** (`git bisect run`).
9. **차분 루프**: 같은 입력을 구버전 vs 신버전에, 출력 diff.
10. 최후의 수단으로 **사람 개입 스크립트** — 수동 단계도 구조화되게.

### 루프 조이기
루프를 제품처럼 다룬다: 더 빠르게(셋업 캐시, 범위 축소), 더 날카롭게(특정 증상 검증, "안 죽었다" 아니라), 더 결정적으로(시간 고정, RNG 시드, FS/네트워크 격리). 2초 결정적 루프는 디버깅 초능력이다.

### 비결정적 버그
목표는 깨끗한 재현이 아니라 높은 재현율이다. 100회 루프, 병렬화, 스트레스 추가, 타이밍 창 좁히기. 1% 플레이크를 디버깅 가능하게 끌어올린다.

이 전술들은 *런타임* 엔트로피(스레드, 타이머, async)를 겨냥한다. 하지만 많은 플레이크는 런타임 엔트로피가 0이다 — 변화가 upstream의 **입력 순서, 해시/이터레이션 순서, 로케일, env**에서 들어온다. 그러니 먼저 묻는다: **실패한 런과 성공한 런 사이에 무엇이 다른가?** 그다음 그 변화를 직접 주입한다. 시드 RNG는 결정성 고정용만이 아니다 — 입력 공간을 탐색하는 *퍼저*가 될 수 있다: 입력의 셔플에 시드를 주고 루프 돌려, 드문 플레이크를 100% red 신호로(그 자체로 재현 가능하게) 바꾼다.

### 완료 기준
이미 **실행해 본 명령 하나**를 댈 수 있으면 Phase 1 완료: red 가능(사용자의 정확한 증상 검증), 결정적, 빠름, 에이전트 실행 가능. **red 가능한 명령이 없으면 Phase 2 없음.** "가끔 / 이따금" 리포트는 흔히 진짜 랜덤이 아니다 — 플레이크를 쫓기 전에 결정적 enumerator(모든 페이지, 모든 크기, 모든 입력 순서를 순회)로 바꾼다.

## Phase 2 — 재현 + 최소화
루프 실행; red 를 본다. **사용자의** 실패가 맞는지 확인(근처의 다른 실패 아님). 그다음 여전히 red 인 가장 작은 시나리오로 줄인다. 입력/호출자/설정을 하나씩 잘라낸다. 남은 모든 요소가 load-bearing 이면 완료.

## Phase 3 — 가설
테스트 전에 **3~5개의 순위 매긴 반증 가능한 가설**을 만든다. 형식: "X 가 원인이면, Y 를 바꾸면 버그가 사라진다." 순위 목록을 사용자에게 보여준다(도메인 지식으로 즉시 재정렬하곤 한다). 자리에 없으면 막지 않는다.

## Phase 4 — 계측
각 프로브는 특정 예측에 매핑된다. 한 번에 한 변수만. 로그보다 디버거/REPL 선호; "전부 로깅"보다 경계 타깃 로그. **모든 디버그 로그에 고유 접두사 태그**(예: `[DEBUG-a4f2]`)로 정리는 grep 한 번. Phase 1의 실패 단언이 이미 결함을 국소화했다면 추가 계측이 필요 없다 — `console.log` 의식은 건너뛴다.

성능은 먼저 측정한다(베이스라인, 프로파일러, 쿼리 플랜). **이분 탐색은 시간에 걸친 회귀나 넓은 표면용(`git bisect`)이다; 국소 핫루프에선 이분 탐색할 대상이 없다 — 대신 프로파일로 핫 프레임을 찾는다.** 단일 크기의 wall-clock이 아니라 여러 크기에 걸친 *요소당*(amortized) 비용을 보고하는 크기 스윕을 써서, 초선형 스케일링이 곡선으로 보이게 한다.

## Phase 5 — 수정 + 회귀 테스트
**수정 전에** 회귀 테스트를 작성하되, 올바른 seam 이 있을 때만(호출 지점에서 실제 버그 패턴을 실행하는 seam). 올바른 seam 이 없으면 **그것 자체가 발견**이다 — 기록한다; 아키텍처가 봉인을 막고 있다. 있으면: 실패 테스트 → 실패 확인 → 수정 → 통과 확인 → 원래(최소화 안 한) 루프 재실행. **성능 수정**의 최대 리스크는 출력이 조용히 바뀌는 것이다 — 타이밍 단언에 **출력 동등성 테스트**(골든 출력, 또는 신·구 구현을 diff 하는 프로퍼티 테스트)를 짝지어 더 빠른 것만이 아니라 동작이 그대로임을 증명한다. **플레이크 버그**는 회귀 테스트가 버그 코드에 대해 *안정적으로* red 인지(구버전에 돌려) 확인한다, 이따금 red 가 아니라.

## Phase 6 — 정리 + 포스트모템
- [ ] 원래 재현이 더는 재현 안 됨
- [ ] 회귀 테스트 통과 (또는 seam 부재 문서화)
- [ ] 모든 `[DEBUG-...]` 계측 제거
- [ ] 일회용 프로토타입 삭제
- [ ] 맞았던 가설을 커밋/PR 에 기술해 다음 디버거가 배우게

그다음 묻는다: **무엇이 이 버그를 막았을까?** 답이 아키텍처적이면(좋은 seam 없음, 얽힌 호출자, 숨은 결합) 수정이 들어간 **후에** `improve-codebase-architecture` 스킬로 구체사항과 함께 넘긴다.

## 오너 / 리더십 렌즈

- **누가 가장 크게 소리쳤는지가 아니라 영향 반경으로 분류한다**: 몇 명의 사용자, 데이터 손상 여부, 되돌릴 수 있는지, 돈/신뢰가 걸렸는지.
- **비난 없는 포스트모템.** 산출물은 범인이 아니라 시스템적 수정(seam, 가드레일, 테스트)이다. 반복되는 버그 클래스는 아키텍처/프로세스 신호다.
- **"충분히 좋음"이 끝일 때를 결정한다.** 창업자는 출시한다; 0.01% 엣지 케이스의 완벽한 재현이 그것에 드는 하루만큼 가치 없을 수 있다. 그 트레이드오프를 명시적으로 한다.
