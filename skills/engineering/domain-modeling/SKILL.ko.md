# 도메인 모델링 (Domain Modeling)

설계하면서 프로젝트의 도메인 모델을 능동적으로 만들고 다듬는다. 이것은 *능동적인* 규율이다 — 용어에 시비를 걸고, 엣지 케이스 시나리오를 지어내고, 결정이 굳어지는 순간 용어집과 결정을 적어 둔다. (어휘를 얻으려고 `CONTEXT.md` 를 *읽기만* 하는 것은 이 스킬이 아니다 — 그건 어느 스킬이나 하는 한 줄짜리 습관이다. 이 스킬은 모델을 소비하는 게 아니라 모델을 바꿀 때를 위한 것이다.)

## 파일 구조

대부분의 레포는 단일 컨텍스트를 가진다:

```
/
├── CONTEXT.md
├── docs/
│   └── adr/
│       ├── 0001-event-sourced-orders.md
│       └── 0002-postgres-for-write-model.md
└── src/
```

루트에 `CONTEXT-MAP.md` 가 있으면 레포에 여러 컨텍스트가 있다. 맵이 각 컨텍스트의 위치를 가리킨다:

```
/
├── CONTEXT-MAP.md
├── docs/
│   └── adr/                          ← 시스템 전역 결정
├── src/
│   ├── ordering/
│   │   ├── CONTEXT.md
│   │   └── docs/adr/                 ← 컨텍스트별 결정
│   └── billing/
│       ├── CONTEXT.md
│       └── docs/adr/
```

파일은 게으르게 만든다 — 적을 게 생겼을 때만. 아직 `CONTEXT.md` 가 없으면 첫 용어가 정리될 때 만든다. 아직 `docs/adr/` 가 없으면 첫 ADR 이 필요할 때 만든다. 빈 파일을 미리 깔아 두지 않는다.

## 세션 중에

### 용어집에 시비 걸기

사용자가 `CONTEXT.md` 의 기존 언어와 충돌하는 용어를 쓰면 즉시 짚는다. "용어집에선 'cancellation' 을 X 로 정의했는데, 지금은 Y 를 뜻하는 것 같습니다 — 어느 쪽인가요?"

### 흐릿한 언어 날카롭게

사용자가 모호하거나 과부하된 용어를 쓰면 정확한 표준 용어를 제안한다. "'account' 라고 하셨는데 — Customer 를 뜻하나요 User 를 뜻하나요? 둘은 다른 것입니다."

### 구체적 시나리오로 논의

도메인 관계를 논의할 때 구체적 시나리오로 압박 테스트한다. 엣지 케이스를 찌르고 개념 간 경계를 정확히 말하게 강제하는 시나리오를 지어낸다.

### 코드와 교차 검증

사용자가 무언가 어떻게 동작한다고 말하면 코드가 동의하는지 확인한다. 모순을 발견하면 드러낸다: "코드는 Order 전체를 취소하는데, 방금 부분 취소가 가능하다고 하셨습니다 — 어느 쪽이 맞나요?"

### CONTEXT.md 즉시 갱신

용어가 정리되면 바로 그 자리에서 `CONTEXT.md` 를 갱신한다. 모아 두지 말고 발생하는 대로 기록한다. [references/CONTEXT-FORMAT.ko.md](references/CONTEXT-FORMAT.ko.md) 의 포맷을 사용한다.

`CONTEXT.md` 에는 구현 디테일이 전혀 없어야 한다. `CONTEXT.md` 를 명세, 메모장, 구현 결정 저장소로 다루지 않는다. 그것은 용어집일 뿐 그 이상이 아니다.

### ADR 은 아껴서 제안

다음 세 가지가 모두 참일 때만 ADR 생성을 제안한다:

1. **되돌리기 어려움** — 나중에 마음을 바꾸는 비용이 의미 있게 크다
2. **맥락 없으면 의외** — 미래의 독자가 "왜 이렇게 했지?" 하고 의아해할 것이다
3. **실제 트레이드오프의 결과** — 진짜 대안들이 있었고 특정 이유로 하나를 골랐다

셋 중 하나라도 빠지면 ADR 을 건너뛴다. [references/ADR-FORMAT.ko.md](references/ADR-FORMAT.ko.md) 의 포맷을 사용한다.

## 오너 / 리더십 렌즈

개발자이자 CEO/CTO/CFO/PM 인 사람에게, 공유된 도메인 언어는 관료주의가 아니라 커뮤니케이션 자산이다:

- **용어집은 정렬 자본이다.** 팀, 투자자, 내 문서가 모두 같은 것을 같은 단어로 부르면 모든 대화가 싸진다. 모호한 용어("account", "user", "subscription")는 실제 돈을 쓴다 — 잘못 만든 기능, 잘못 잡은 계약 범위, 다시 설명하는 데 쓰는 온보딩 시간으로.
- **ADR 은 조직의 기억이다.** 되돌릴 수 없는 결정 뒤의 이유는 결정 자체보다 가치 있다. 6개월 뒤 기록된 "왜" 가 당신(또는 신입, 또는 에이전트)이 이미 끝난 트레이드오프를 다시 다투거나 의도적이었던 것을 "고치는" 일을 막는다.
- **언어가 곧 제품의 경계다.** Customer 와 User, 또는 Order 와 Invoice 사이에 선을 어디 긋느냐는 스키마이기 전에 비즈니스 결정이다. 단어를 제대로 잡으면 계약, 가격, 리포팅이 거기서 흘러나온다.
