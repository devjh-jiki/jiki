# 코드베이스 아키텍처 개선

아키텍처 마찰을 드러내고 **깊게 만들 기회(deepening opportunities)** — 얕은 모듈을 깊은 모듈로 바꾸는 리팩터링 — 를 제안한다. 목표는 테스트 용이성과 AI 탐색 용이성.

이 스킬은 공유 설계 어휘 위에 세워진다:

- 아키텍처 어휘(**module**, **interface**, **depth**, **seam**, **adapter**, **leverage**, **locality**)와 그 원칙(삭제 테스트, "인터페이스가 테스트 표면이다", "어댑터 하나 = 가설적 seam, 둘 = 진짜")을 위해 `codebase-design` 스킬을 실행하라. 모든 제안에서 이 용어들을 정확히 써라 — "component", "service", "API", "boundary" 로 흘러가지 마라.
- 프로젝트에 컨텍스트 문서(예: `CONTEXT.md`)나 아키텍처 결정 기록이 있으면 먼저 읽어라: 프로젝트의 도메인 언어가 좋은 seam 에 이름을 주고, 기록된 결정은 이 스킬이 다시 들쑤시지 말아야 할 것들이다.

## 절차

### 1. 탐색

건드릴 영역에 대한 프로젝트 컨텍스트 문서(예: `CONTEXT.md`)와 기록된 아키텍처 결정이 있으면 먼저 읽어라.

그다음 Agent 도구를 `subagent_type=Explore` 로 써서 코드베이스를 걸어라. 경직된 휴리스틱을 따르지 마라 — 유기적으로 탐색하고 마찰을 느끼는 곳을 메모하라:

- 한 개념을 이해하는 데 작은 모듈 여럿을 오가야 하는 곳은?
- 모듈이 **얕은** 곳 — 인터페이스가 구현만큼 복잡한 곳은?
- 순수 함수가 오직 테스트 용이성 때문에 추출됐지만 진짜 버그는 그것이 어떻게 호출되는지에 숨어 있는(지역성 없음) 곳은?
- 강하게 결합된 모듈이 seam 너머로 새는 곳은?
- 테스트가 없거나 현재 인터페이스로는 테스트하기 어려운 부분은?

얕다고 의심되는 것에 **삭제 테스트**를 적용하라: 삭제하면 복잡함이 모이나, 아니면 그냥 옮겨지나? "그렇다, 모인다" 가 원하는 신호다.

### 2. 후보를 HTML 리포트로 제시

레포에 아무것도 남지 않도록 OS 임시 디렉토리에 자체 포함된 HTML 파일을 써라. 임시 디렉토리는 `$TMPDIR` 에서 해결하고 `/tmp`(Windows 는 `%TEMP%`)로 폴백하며, 매 실행이 새 파일을 갖도록 `<tmpdir>/architecture-review-<timestamp>.html` 에 써라. 사용자를 위해 열어라 — Linux 는 `xdg-open <path>`, macOS 는 `open <path>`, Windows 는 `start <path>` — 그리고 절대 경로를 알려줘라.

리포트는 레이아웃·스타일에 **CDN 의 Tailwind**, 그래프/플로우/시퀀스가 구조를 확실히 전달하는 다이어그램에 **CDN 의 Mermaid** 를 쓴다. Mermaid 와 손으로 만든 CSS/SVG 비주얼을 섞어라 — 관계가 그래프 모양일 때(콜 그래프, 의존성, 시퀀스) Mermaid 를, 좀 더 편집적인 것을 원할 때(질량 다이어그램, 단면도, 붕괴 애니메이션) 손으로 만든 div/SVG 를. 각 후보는 **전/후 시각화**를 갖는다. 시각적으로.

각 후보에 대해 카드를 렌더링하라:

- **Files** — 어떤 파일/모듈이 관련되는가
- **Problem** — 현재 아키텍처가 왜 마찰을 일으키는가
- **Solution** — 무엇이 바뀔지 평이한 말로 설명
- **Benefits** — 지역성과 레버리지의 관점, 그리고 테스트가 어떻게 나아지는지로 설명
- **Before / After diagram** — 나란히, 직접 그려서, 얕음과 깊어짐을 보여줌
- **Recommendation strength** — `Strong`, `Worth exploring`, `Speculative` 중 하나, 배지로 렌더링

리포트는 **Top recommendation** 섹션으로 끝낸다: 어떤 후보를 먼저 다룰지, 왜.

**도메인에는 프로젝트 자신의 도메인 언어를, 아키텍처에는 `codebase-design` 어휘를 써라.** 프로젝트 컨텍스트 문서가 "Order" 를 정의하면 "the Order intake module" 이라 말하라 — "the FooBarHandler" 도, "the Order service" 도 아니다.

**기록된 결정과의 충돌**: 후보가 기록된 아키텍처 결정과 모순될 때, 그 결정을 다시 볼 만큼 마찰이 진짜일 때만 드러내라. 카드에 명확히 표시하라(예: 경고 콜아웃: _"X 에 대한 기존 결정과 모순됨 — 그래도 다시 열 가치가 있는 이유는…"_). 과거 결정이 금지하는 모든 이론적 리팩터링을 나열하지 마라.

(위 HTML 골격은 일부러 인라인으로 기술해 유지할 별도 참조 파일이 없게 했다 — CDN Tailwind + Mermaid 와 후보당 카드 하나로 된 자체 포함 단일 파일이 명세 전부다.)

아직 인터페이스를 제안하지 마라. 파일을 쓴 뒤 사용자에게 물어라: "이 중 어느 것을 탐색하고 싶으세요?"

### 3. 압박 질문 루프

사용자가 후보를 고르면 `grill-me` 스킬을 실행해 설계 트리를 함께 걸어라 — 제약, 의존성, 깊어진 모듈의 모양, seam 뒤에 무엇이 있는지, 어떤 테스트가 살아남는지.

결정이 굳어지면서 부수효과가 인라인으로 일어난다. 프로젝트가 컨텍스트 문서(예: `CONTEXT.md`)나 결정 로그를 유지하면, 진행하며 최신으로 유지하라:

- **깊어진 모듈을 프로젝트 컨텍스트 문서에 아직 없는 개념으로 명명하나?** 그 용어를 추가하라. 문서가 없고 프로젝트가 원하면 그때 만들어라.
- **대화 중 모호한 용어를 날카롭게 다듬었나?** 바로 그 자리에서 컨텍스트 문서를 갱신하라.
- **사용자가 하중을 견디는 이유로 후보를 거절했나?** 기록을 제안하라: _"미래의 아키텍처 리뷰가 이걸 다시 제안하지 않도록 적어둘까요?"_ 미래의 탐색자가 같은 것을 다시 제안하지 않으려면 실제로 필요할 이유일 때만 제안하라 — 일시적 이유("지금은 가치 없음")와 자명한 것은 건너뛰어라.
- **깊어진 모듈의 대안 인터페이스를 탐색하고 싶나?** `codebase-design` 스킬을 실행하고 그 두 번 설계하기 병렬 서브에이전트 패턴을 써라.

## 오너 / 리더십 렌즈

CEO/CTO/CFO/PM 를 겸하는 개발자에게 아키텍처 리뷰는 포트폴리오 리뷰다:

- **가려움이 아니라 ROI 로 순위를 매겨라.** 리포트의 추천 강도는 예산 도구다 — `Strong` 은 엔지니어링 시간이 복리로 쌓이는 곳, `Speculative` 는 베팅이다. 매출을 풀어주거나 반복 유지비를 줄이는 리팩터링에 자금을 대고, 나머지는 미뤄라.
- **깊게 만들기를 비즈니스 리스크와 묶어라.** 벤더 seam, 결제 경로, 인증 경계의 얕은 모듈은 집중된 리스크다. 틀리면 돈·신뢰·재작성이 드는 seam 을 우선하라.
- **아키텍처 부채는 대차대조표 항목이다.** 보이게 하라: 이 리포트는 공동창업자나 이사회에 "부채 상환" 스프린트를 정당화하기 위해 보여줄 수 있는 것 — 여느 자본 지출을 정당화하듯이.
