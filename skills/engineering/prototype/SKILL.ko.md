# 프로토타입 (Prototype)

프로토타입은 **질문에 답하는 버릴 코드**다. 질문이 형태를 결정한다.

## 가지 고르기

어떤 질문에 답하는지 식별한다 — 사용자의 프롬프트, 주변 코드, 또는 사용자가 곁에 있으면 물어서:

- **"이 로직 / 상태 모델이 맞는 느낌인가?"** → [references/LOGIC.md](references/LOGIC.ko.md). 종이 위에선 추론하기 어려운 케이스들로 상태 기계를 밀어붙이는 작은 인터랙티브 터미널 앱을 만든다.
- **"이게 어떻게 생겨야 하나?"** → [references/UI.md](references/UI.ko.md). 한 라우트에 근본적으로 다른 UI 변형 여러 개를 만들고, URL 검색 파라미터와 떠 있는 하단 바로 전환한다.

두 가지는 매우 다른 산출물을 낳는다 — 잘못 고르면 프로토타입 전체가 낭비된다. 질문이 진짜 애매하고 사용자에게 닿을 수 없으면, 주변 코드에 더 맞는 가지를 기본으로 한다(백엔드 모듈 → 로직; 페이지나 컴포넌트 → UI) 그리고 프로토타입 맨 위에 그 가정을 명시한다.

## 양쪽 모두에 적용되는 규칙

1. **첫날부터 버릴 것, 그리고 그렇게 명확히 표시.** 프로토타입 코드를 실제로 쓰일 곳 가까이(프로토타이핑 대상 모듈이나 페이지 옆) 둬서 맥락이 분명하게 — 그러나 무심한 독자도 프로덕션이 아니라 프로토타입임을 알 수 있게 이름 짓는다. 버릴 UI 라우트는 프로젝트가 이미 쓰는 라우팅 관례를 따른다; 새 최상위 구조를 발명하지 않는다.
2. **한 명령으로 실행.** 프로젝트의 기존 태스크 러너가 지원하는 것이면 무엇이든 — `pnpm <name>`, `python <path>`, `bun <path>` 등. 사용자가 생각 없이 시작할 수 있어야 한다.
3. **기본적으로 영속성 없음.** 상태는 메모리에 산다. 영속성은 프로토타입이 _확인하는_ 대상이지 의존해야 할 것이 아니다. 질문이 명시적으로 데이터베이스를 포함하면 스크래치 DB 나 "PROTOTYPE — wipe me" 같은 분명한 이름의 로컬 파일을 친다.
4. **다듬기는 건너뛴다.** 테스트 없음, 프로토타입을 _실행 가능_하게 하는 것 이상의 에러 처리 없음, 추상화 없음. 핵심은 빠르게 뭔가 배우고 지우는 것이다.
5. **상태를 드러낸다.** 매 액션 후(로직) 또는 매 변형 전환 시(UI), 관련 상태 전체를 출력하거나 렌더해서 사용자가 무엇이 바뀌었는지 보게 한다.
6. **끝나면 지우거나 흡수한다.** 프로토타입이 질문에 답했으면, 지우거나 검증된 결정을 실제 코드에 접어 넣는다 — 레포에 썩게 두지 않는다.

## 끝났을 때

프로토타입에서 간직할 가치가 있는 것은 _답_ 뿐이다. 답하던 질문과 함께 지속되는 곳(커밋 메시지, ADR, 이슈, 또는 프로토타입 옆 `NOTES.md`)에 잡아 둔다. 사용자가 곁에 있으면 그 기록은 짧은 대화고; 아니면 자리표시자를 남겨 그들(또는 다음 회차의 당신)이 프로토타입을 지우기 전에 결론을 채울 수 있게 한다.

## 오너 / 리더십 렌즈

개발자이자 CEO/CTO/CFO/PM 인 사람에게, 버릴 프로토타입은 리스크 관리 도구다:

- **프로토타입은 빌드 예산을 쓰기 전에 결정의 리스크를 줄이는 가장 싼 방법이다.** 나쁜 UI 방향을 죽이거나 망가진 상태 모델을 드러내는 하루치 일회용 코드가, 잘못된 방향으로 향한 한 스프린트의 실제 구현을 아낀다.
- **답이 자산이고, 코드는 일회용이다.** 산출물에 애착을 갖지 마라. 당신이 사는 것은 확신 — "해봤고, 이걸 배웠다" — 이지 간직할 코드 줄 수가 아니다.
- **시간 상자를 둬라.** 테스트, 추상화, 영속성이 자라난 프로토타입은 예산에 없던 프로덕션 작업이 슬그머니 된 것이다. 버리는 규율이 그것을 싸게 유지한다.
