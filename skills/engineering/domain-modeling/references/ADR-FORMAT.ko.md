# ADR 포맷

ADR 은 `docs/adr/` 에 두고 순차 번호를 쓴다: `0001-slug.md`, `0002-slug.md` 등.

`docs/adr/` 디렉토리는 게으르게 만든다 — 첫 ADR 이 필요할 때만.

## 템플릿

```md
# {결정의 짧은 제목}

{1~3문장: 맥락은 무엇이고, 무엇을 결정했고, 왜.}
```

이게 전부다. ADR 은 한 문단일 수 있다. 가치는 결정이 내려졌다는 *사실*과 그 *이유*를 기록하는 데 있다 — 섹션을 채우는 데 있지 않다.

## 선택 섹션

진짜 가치를 더할 때만 포함한다. 대부분의 ADR 은 필요 없다.

- **Status** frontmatter (`proposed | accepted | deprecated | superseded by ADR-NNNN`) — 결정이 다시 검토될 때 유용
- **Considered Options** — 기각된 대안을 기억할 가치가 있을 때만
- **Consequences** — 자명하지 않은 하류 영향을 짚어야 할 때만

## 번호 매기기

`docs/adr/` 를 훑어 가장 큰 기존 번호를 찾고 1 을 더한다.

## ADR 을 제안할 때

다음 세 가지가 모두 참이어야 한다:

1. **되돌리기 어려움** — 나중에 마음을 바꾸는 비용이 의미 있게 크다
2. **맥락 없으면 의외** — 미래의 독자가 코드를 보며 "대체 왜 이렇게 했지?" 하고 의아해할 것이다
3. **실제 트레이드오프의 결과** — 진짜 대안들이 있었고 특정 이유로 하나를 골랐다

되돌리기 쉬운 결정이면 건너뛴다 — 어차피 되돌릴 거다. 의외가 아니면 아무도 왜냐고 묻지 않는다. 진짜 대안이 없었다면 "당연한 걸 했다" 외에 기록할 게 없다.

### 자격이 되는 것

- **아키텍처 형태.** "모노레포를 쓴다." "쓰기 모델은 이벤트 소싱, 읽기 모델은 Postgres 로 투영한다."
- **컨텍스트 간 통합 패턴.** "Ordering 과 Billing 은 동기 HTTP 가 아니라 도메인 이벤트로 통신한다."
- **락인을 동반하는 기술 선택.** 데이터베이스, 메시지 버스, 인증 제공자, 배포 타겟. 모든 라이브러리가 아니라 — 교체에 한 분기는 걸릴 것들만.
- **경계와 범위 결정.** "Customer 데이터는 Customer 컨텍스트가 소유하고, 다른 컨텍스트는 ID 로만 참조한다." 명시적인 "안 한다"는 "한다"만큼 가치 있다.
- **자명한 길에서의 의도적 이탈.** "X 때문에 ORM 대신 수동 SQL 을 쓴다." 합리적인 독자가 반대를 가정할 만한 모든 것. 다음 엔지니어가 의도적이었던 것을 "고치는" 일을 막는다.
- **코드에 보이지 않는 제약.** "컴플라이언스 요구 때문에 AWS 를 못 쓴다." "파트너 API 계약 때문에 응답 시간이 200ms 미만이어야 한다."
- **기각이 자명하지 않은 기각된 대안.** GraphQL 을 검토했다가 미묘한 이유로 REST 를 골랐다면 기록한다 — 안 그러면 6개월 뒤 누군가 또 GraphQL 을 제안한다.
