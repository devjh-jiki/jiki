# 코드베이스 설계

**깊은 모듈(deep module)**을 설계하라: 작은 인터페이스 뒤에 많은 동작을 두고, 깔끔한 seam 에 배치하며, 그 인터페이스를 통해 테스트한다. 코드를 설계하거나 재구성하는 모든 곳에서 이 언어와 원칙을 쓴다. 목표는 호출자에게는 레버리지, 유지보수자에게는 지역성(locality), 모두에게는 테스트 용이성.

이 스킬은 *한 모듈의 모양과 인터페이스* 규모에서 작동한다. 두 가지로 발동한다: 누군가 단일 모듈을 설계하거나 판단할 때 단독으로, 그리고 다른 스킬이 가져다 쓰는 공유 어휘로서(예: `improve-codebase-architecture` 가 재구조화하며 이 용어들을 쓴다). 두 경계가 이 스킬을 제 자리에 둔다: 아직 구체적인 모듈이 시야에 없고 막연히 "아키텍처 개선"만 있으면, 어휘를 적용하기 전에 어느 모듈/seam 이 문제인지 먼저 짚어라, 없는 걸 지어내지 말고. 그리고 실제 과제가 기존 코드베이스의 여러 모듈을 대규모로 재구조화하는 것(여러 seam 에 걸친 deepening 기회 매핑, 작업 순서 정하기)이라면 `improve-codebase-architecture` 로 넘겨라 — 그쪽이 같은 언어로 적정 규모에서 그 일을 끌고 간다.

## 용어집

이 용어들을 정확히 써라 — "component", "service", "API", "boundary" 로 바꾸지 마라. 일관된 언어가 핵심 그 자체다.

**Module** — 인터페이스와 구현을 가진 모든 것. 의도적으로 규모 무관: 함수, 클래스, 패키지, 또는 계층을 가로지르는 슬라이스. _피할 것_: unit, component, service.

**Interface** — 호출자가 모듈을 올바르게 쓰기 위해 알아야 하는 모든 것: 타입 시그니처뿐 아니라 불변식, 순서 제약, 에러 모드, 필요한 설정, 성능 특성까지. _피할 것_: API, signature (너무 좁음 — 타입 수준 표면만 가리킴).

**Implementation** — 모듈 내부, 그 코드 본문. **Adapter** 와 구분: 작은 어댑터에 큰 구현(Postgres 리포)일 수도, 큰 어댑터에 작은 구현(인메모리 페이크)일 수도 있다. seam 이 주제일 때 "adapter", 그 외에는 "implementation".

**Depth** — 인터페이스에서의 레버리지: 호출자(또는 테스트)가 배워야 하는 인터페이스 단위당 행사할 수 있는 동작의 양. 작은 인터페이스 뒤에 많은 동작이 있으면 **깊고(deep)**, 인터페이스가 구현만큼 복잡하면 **얕다(shallow)**.

**Seam** _(Michael Feathers)_ — 그 자리를 편집하지 않고도 동작을 바꿀 수 있는 곳; 모듈의 인터페이스가 사는 *위치*. seam 을 어디에 둘지는 그 뒤에 무엇이 가는지와 별개인 설계 결정이다. _피할 것_: boundary (DDD 의 bounded context 와 의미가 겹침).

**Adapter** — seam 에서 인터페이스를 만족하는 구체적인 것. 실체(내부에 뭐가 있는지)가 아니라 *역할*(어떤 슬롯을 채우는지)을 가리킨다.

**Leverage** — 깊이에서 호출자가 얻는 것: 배우는 인터페이스 단위당 더 많은 능력. 하나의 구현이 N 개의 호출 지점과 M 개의 테스트에 걸쳐 보답한다.

**Locality** — 깊이에서 유지보수자가 얻는 것: 변경, 버그, 지식, 검증이 호출자들에 흩어지지 않고 한 곳에 모인다. 한 번 고치면 모두 고쳐진다.

## 깊음 vs 얕음

**깊은 모듈** = 작은 인터페이스 + 많은 구현:

```
┌─────────────────────┐
│   Small Interface   │  ← 메서드 적음, 단순한 파라미터
├─────────────────────┤
│                     │
│  Deep Implementation│  ← 복잡한 로직 숨김
│                     │
└─────────────────────┘
```

**얕은 모듈** = 큰 인터페이스 + 적은 구현 (피하라):

```
┌─────────────────────────────────┐
│       Large Interface           │  ← 메서드 많음, 복잡한 파라미터
├─────────────────────────────────┤
│  Thin Implementation            │  ← 그냥 통과시킴
└─────────────────────────────────┘
```

인터페이스를 설계할 때 물어라:

- 메서드 수를 줄일 수 있나?
- 파라미터를 단순화할 수 있나?
- 복잡함을 더 안으로 숨길 수 있나?

## 원칙

- **깊이는 구현이 아니라 인터페이스의 속성이다.** 깊은 모듈은 내부적으로 작고 mock 가능하고 교체 가능한 부품들로 이뤄질 수 있다 — 그것들이 인터페이스의 일부가 아닐 뿐. 모듈은 인터페이스의 **외부 seam** 뿐 아니라 **내부 seam**(구현에 사적이고 자기 테스트가 쓰는)도 가질 수 있다.
- **삭제 테스트.** 모듈을 삭제한다고 상상하라. 복잡함이 사라지면 통과용(pass-through)이었던 것. 복잡함이 N 개 호출자에 다시 나타나면 제 몫을 하던 것. *인터페이스*와 *구현*에 따로 적용하라: 한 멤버짜리 `RetryPolicy` 인터페이스는 테스트에 떨어질 수 있어도(삭제) 그 뒤의 backoff *로직*은 통과할 수 있다(유지). 그리고 삭제의 결과가 그 모듈이 진짜 정책을 보유하는지에 달려 있을 때(그냥 전달만 할 수도, fan-out/rate-limiting 을 소유할 수도 있는 dispatcher), 판단 전에 그 동작을 들여다보라 — 호출자 수만으로는 알 수 없다.
- **인터페이스가 테스트 표면이다.** 호출자와 테스트는 같은 seam 을 건넌다. 인터페이스 *너머*를 테스트하고 싶다면 모듈 모양이 잘못된 것이다.
- **어댑터 하나면 가설적 seam. 둘이면 진짜 seam.** 무언가 실제로 그 너머에서 달라지지 않는 한 seam 을 도입하지 마라. *구현의 개수*가 아니라 *변이의 축*을 세라: 한 인터페이스의 N 개 구현(하나의 `Channel` 뒤의 알림 채널 세 개)은 여전히 정확히 seam 하나를 정당화하지, N 개가 아니다.
- **깊이는 샐 수 있다.** 깊이는 인터페이스에서의 레버리지다 — 하지만 제어를 역전시키는 인터페이스(호출자가 준 콜백/핸들러를 받는 스트리밍 모듈)는 동작상 깊으면서도 복잡도를 호출자에게 *수출*할 수 있다(콜백 순서, 재진입). 그 수출된 복잡도를 인터페이스에 합산하라; 작은 타입 시그니처가 자동으로 작은 인터페이스인 건 아니다.

## 테스트 용이성을 위한 설계

좋은 인터페이스는 테스트를 자연스럽게 만든다:

1. **의존성을 만들지 말고 받아라.**

   ```typescript
   // 테스트 가능
   function processOrder(order, paymentGateway) {}

   // 테스트 어려움
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **부수효과를 내지 말고 결과를 반환하라.**

   ```typescript
   // 테스트 가능
   function calculateDiscount(cart): Discount {}

   // 테스트 어려움
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **작은 표면적.** 메서드가 적을수록 필요한 테스트가 적다. 파라미터가 적을수록 테스트 셋업이 단순하다.

## 관계

- **Module** 은 정확히 하나의 **Interface** 를 가진다 (호출자와 테스트에 보이는 표면).
- **Depth** 는 **Module** 의 속성이며, 그 **Interface** 에 대해 측정된다.
- **Seam** 은 **Module** 의 **Interface** 가 사는 곳이다.
- **Adapter** 는 **Seam** 에 앉아 **Interface** 를 만족한다.
- **Depth** 는 호출자에게 **Leverage**, 유지보수자에게 **Locality** 를 만든다.

## 기각된 프레이밍

- **구현 줄 수 대 인터페이스 줄 수의 비율로서의 깊이** (Ousterhout): 구현을 부풀리는 걸 보상한다. 우리는 레버리지로서의 깊이를 쓴다.
- **TypeScript `interface` 키워드나 클래스의 public 메서드로서의 "interface"**: 너무 좁음 — 여기서 인터페이스는 호출자가 알아야 할 모든 사실을 포함한다.
- **"Boundary"**: DDD 의 bounded context 와 겹침. **seam** 또는 **interface** 라고 하라.

## 더 깊이 (선택)

별도 참조 파일 없이 인라인으로 요약한 두 가지 심화 동작:

- **의존성을 고려해 클러스터 깊게 만들기** — 모듈의 의존성을 분류하고, seam 규율을 지키며(인터페이스를 통해 테스트하고 너머로 손을 뻗지 말 것), 또 다른 얕은 래퍼를 *덧씌우기*보다 어댑터를 *교체*하는 쪽을 선호하라. 프로젝트 컨텍스트 문서(예: `CONTEXT.md`)가 있으면 거기 정의된 개념 이름으로 모듈을 명명하라.
- **두 번 설계하기** — 인터페이스를 확정하기 전에 근본적으로 다른 여러 방식으로 스케치하고(병렬 서브에이전트를 띄워 할 수 있다), 깊이·지역성·seam 위치로 후보들을 비교하라. 승자는 첫 스케치인 경우가 드물다. 두 가지를 분명히: 후보가 "근본적으로 다르다"고 치려면 **seam 이 다른 곳에** 있어야 한다(다른 인터페이스, 건너는 것이 다름) — 겉만 다른 변형은 안 된다; ~3개를 목표로. 그리고 발산적 스케치는 seam 을 *발명하도록* 유혹한다 — 그러니 어댑터 하나/둘 규칙과 삭제 테스트를 스케치 *도중*이 아니라 **이후에 필터로** 적용하라. 먼저 발산, 그다음 가지치기.

## 오너 / 리더십 렌즈

CEO/CTO/CFO/PM 를 겸하는 개발자에게 모듈 설계는 자본 배분 결정이다:

- **깊이는 레버리지이고, 레버리지는 복리로 쌓인다.** 깊은 모듈은 모든 호출자와 모든 미래 기능에 걸쳐 보답한다 — 좋은 채용이나 재사용 가능한 프로세스와 똑같다. 보답이 가장 큰 곳에 설계 노력을 써라.
- **지역성은 버스 팩터와 온보딩 비용을 낮춘다.** 변경이 한 곳에 모이면 새 팀원(또는 미래의 당신, 또는 에이전트)이 시스템 전체를 머릿속에 담지 않고도 생산적일 수 있다. 인터페이스 설계로 통제하는 실질 조직 비용이다.
- **Seam 은 선택권(optionality)이다.** 깔끔한 seam 은 나중에 벤더를 바꾸거나, 가격 엔진을 교체하거나, 느린 경로를 재작성 없이 갈아끼우게 해 준다 — 단일 공급사에 묶이지 않는 것의 엔지니어링 판본이다. 하지만 선택권엔 프리미엄이 있다: 투기적 seam(어댑터 하나, 로드맵에 두 번째 없음)은 더 많은 간접화·낮은 지역성·온보딩 비용으로 *매일* 값을 치르는 반면, "나중에 인터페이스 추출"은 두 번째 어댑터가 실제로 생긴 뒤 완전한 정보로 하는 싸고 거의 기계적인 리팩터다. 옵션은 두 번째 어댑터가 손에 있을 때 사라, 스펙만 보고 6개월 일찍 사지 마라.
