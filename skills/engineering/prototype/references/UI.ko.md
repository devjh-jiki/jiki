# UI 프로토타입 (UI Prototype)

한 라우트에 **근본적으로 다른 UI 변형 여러 개**를 만들고, 떠 있는 하단 바로 전환한다. 사용자가 브라우저에서 변형들을 넘겨 보고, 하나를 고르거나(또는 각각에서 일부를 훔치거나), 나머지는 버린다.

질문이 무엇처럼 보이느냐가 아니라 로직/상태에 관한 것이면 — 틀린 가지. [LOGIC.md](LOGIC.ko.md) 를 사용한다.

## 이 형태가 맞을 때

- "이 페이지가 어떻게 생겨야 하나?"
- "확정하기 전에 이 대시보드의 옵션 몇 개를 보고 싶다."
- "설정 화면의 다른 레이아웃을 시도해 봐."
- 사용자가 안 그러면 머릿속 흐릿한 목업 셋 중 고르는 데 하루를 쓸 모든 때.

## 두 하위 형태 — 하위 형태 A 를 강하게 선호

UI 프로토타입은 **앱의 나머지에 맞붙어 있을 때** 판단하기 훨씬 쉽다 — 실제 헤더, 실제 사이드바, 실제 데이터, 실제 밀도. 혼자 떠 있는 버릴 라우트는 진공이다: 격리된 상태에선 모든 변형이 괜찮아 보인다. 변형을 호스팅할 그럴듯한 기존 페이지가 있으면 언제나 하위 형태 A 를 기본으로 한다. 프로토타입에 진짜 가까운 집이 없을 때만 하위 형태 B 로 간다.

### 하위 형태 A — 기존 페이지에 대한 조정 (선호)

라우트가 이미 존재한다. 변형은 `?variant=` URL 검색 파라미터로 게이트되어 **같은 라우트에** 렌더된다. 기존 데이터 페칭, 파라미터, 인증은 모두 그대로 — 렌더링만 바뀐다. 이게 기본이다; 그러지 않을 구체적 이유가 없으면 이걸 고른다.

아직 페이지가 없지만 *자연스럽게 어떤 페이지 안에 살 것*(대시보드의 새 섹션, 설정 화면의 새 카드, 기존 흐름의 새 단계)을 프로토타이핑한다면 — 그것도 여전히 하위 형태 A 다. 변형을 호스트 페이지 안에 마운트한다.

### 하위 형태 B — 새 페이지 (최후의 수단)

프로토타이핑 대상이 진짜로 안에 살 기존 페이지가 없을 때만 쓴다 — 예: 완전히 새로운 최상위 표면, 또는 어디에도 합리적으로 끼워 넣을 수 없는 흐름.

프로젝트가 이미 쓰는 라우팅 관례를 따라 **버릴 라우트**를 만든다 — 새 최상위 구조를 발명하지 않는다. 명백히 프로토타입임을 알 수 있게 이름 짓는다(예: 경로나 파일명에 `prototype` 단어 포함). 같은 `?variant=` 패턴.

하위 형태 B 에 확정하기 전에 점검한다: 이걸 끼워 넣을 기존 페이지가 정말 없는가? 빈 라우트는 채워진 라우트라면 드러낼 설계 문제를 감춘다.

두 하위 형태 모두 떠 있는 하단 바는 동일하다.

## 과정

### 1. 질문을 명시하고 N 을 고른다

**3개 변형**을 기본으로. 5개를 넘으면 근본적으로 다르지 않고 소음이 되기 시작한다 — 거기서 상한.

계획을 한 줄로, 프로토타입 위치나 파일 맨 위 주석에 적는다:

> "설정 페이지의 세 변형, 기존 `/settings` 라우트에서 `?variant=` 로 전환."

이건 사용자가 반박하러 있든 없든 작동한다.

### 2. 근본적으로 다른 변형을 만든다

각 변형을 초안 잡는다. 각각을 다음에 맞춘다:

- 페이지의 목적과 접근 가능한 데이터.
- 프로젝트의 컴포넌트 라이브러리 / 스타일링 시스템(TailwindCSS, shadcn, MUI, 순수 CSS 등).
- 명확한 export 컴포넌트 이름, 예: `VariantA`, `VariantB`, `VariantC`.

변형은 **구조적으로 달라야** 한다 — 다른 레이아웃, 다른 정보 위계, 다른 주요 어포던스, 색만 다른 게 아니라. 살짝 손본 카드 그리드 셋은 UI 프로토타입이 아니라 벽지다. 두 초안이 너무 비슷하게 나오면 하나를 "카드 그리드 쓰지 마" 라는 명시적 지침으로 다시 한다.

### 3. 엮는다

라우트에 단일 스위처 컴포넌트를 만든다:

```tsx
// 의사 코드 — 프로젝트 프레임워크에 맞춘다
const variant = searchParams.get('variant') ?? 'A';
return (
  <>
    {variant === 'A' && <VariantA {...data} />}
    {variant === 'B' && <VariantB {...data} />}
    {variant === 'C' && <VariantC {...data} />}
    <PrototypeSwitcher variants={['A','B','C']} current={variant} />
  </>
);
```

하위 형태 A(기존 페이지): 기존 데이터 페칭을 모두 스위처 위에 둔다; 변형마다 렌더되는 서브트리만 바뀐다.

하위 형태 B(새 페이지): `/prototype/<name>` 아래 버릴 라우트가 같은 스위처를 마운트한다.

### 4. 떠 있는 스위처를 만든다

화면 하단 중앙에 고정된 작은 바, 세 조각:

- **왼쪽 화살표** — 이전 변형으로 순환(되감김).
- **변형 레이블** — 현재 변형 키, 그리고 변형이 이름을 export 하면 그 이름도 표시. 예: `B — Sidebar layout`.
- **오른쪽 화살표** — 앞으로 순환(되감김).

동작:

- 화살표 클릭은 URL 검색 파라미터를 갱신한다(프레임워크의 라우터 사용 — Next 는 `router.replace`, React Router 는 `navigate` 등) 그래서 변형이 공유 가능하고 새로고침에 안정적이다.
- 키보드: `←` 와 `→` 화살표 키도 순환한다. `<input>`, `<textarea>`, `[contenteditable]` 가 포커스됐을 때는 화살표 키를 가로채지 않는다.
- 페이지와 시각적으로 구별되게(예: 고대비 알약, 은은한 그림자) 해서 평가 중인 설계의 일부가 아님이 명백하게.
- 프로덕션 빌드에서 숨김 — `process.env.NODE_ENV !== 'production'` 또는 동등한 검사로 게이트해서, 떠도는 프로토타입 머지가 바를 사용자에게 출시하지 못하게.

스위처를 단일 공유 컴포넌트에 둬서 두 하위 형태가 재사용하게 한다. 프로젝트에서 공유 UI 가 사는 곳에 둔다.

### 5. 넘긴다

URL(그리고 `?variant=` 키들)을 드러낸다. 사용자가 시간 날 때 넘겨 본다. 흥미로운 피드백은 보통 **"B 의 헤더에 C 의 사이드바를 원해"** — 그게 그들이 진짜 원하는 설계다.

### 6. 답을 잡고 정리한다

한 변형이 이겼으면, 어느 것이고 왜인지 적는다(커밋 메시지, ADR, 이슈, 또는 자리 비우고 돌리는 중이고 사용자가 아직 답 안 했으면 프로토타입 옆 `NOTES.md`). 그다음:

- **하위 형태 A** — 진 변형들과 스위처를 지운다; 승자를 기존 페이지에 접어 넣는다.
- **하위 형태 B** — 이긴 변형을 실제 라우트로 승격하고, 버릴 라우트와 스위처를 지운다.

변형 컴포넌트나 스위처를 널브러뜨려 두지 마라. 빠르게 썩고 다음 독자를 혼란스럽게 한다.

## 안티패턴

- **색이나 문구만 다른 변형.** 그건 손질이지 프로토타입이 아니다. 진짜 변형은 구조에 대해 의견이 다르다.
- **변형 간 코드를 너무 많이 공유.** 공유 `<Header>` 는 괜찮다; 공유 `<Layout>` 은 핵심을 무너뜨린다. 각 변형은 레이아웃을 버릴 자유가 있어야 한다.
- **변형을 실제 뮤테이션에 엮기.** 읽기 전용 프로토타입이면 된다. 변형이 변형(mutate)해야 하면 스텁을 가리킨다 — 질문은 "이게 어떻게 생겨야 하나" 지 "백엔드가 작동하나" 가 아니다.
- **프로토타입을 곧장 프로덕션으로 승격.** 변형 코드는 프로토타입 제약(테스트 없음, 최소 에러 처리) 아래 쓰였다. 접어 넣을 때 제대로 다시 쓴다.
