# to-prd

이 스킬은 현재 대화 맥락과 코드베이스 이해를 받아 PRD 를 만든다. 사용자를 인터뷰하지 마라 — 이미 아는 것을 종합만 하라.

**`product-spec-builder` 와의 관계:** `to-prd` 는 이 대화에서 사고가 이미 끝났다고 가정하고 그것을 *적기만* 한다 — 질문 없음. `product-spec-builder` 는 반대다: 초안 전에 빈틈을 메우려고 *먼저 인터뷰*한다. 큰 미지수가 있다면 이 스킬이 아니라 `product-spec-builder` 가 맞다.

"큰 미지수 vs 빈틈" 리트머스: Problem Statement, 주 사용자, 성공 기준을 *지어내야* 한다면 그건 큰 미지수다 — 멈추고 `product-spec-builder` 로 넘겨라. 틀이 정해졌고 디테일만 열려 있으면(기본값, 정확한 문구, 부차적 엣지 케이스) 그건 빈틈이다 — Further Notes 에 미해결 질문으로 적고 그냥 종합하라. 리다이렉트 자체가 체크포인트다: 왜인지 말하고, 사용자가 사실 답을 갖고 있는데 안 적은 것뿐이면 진행하겠다고 제안하라.

## 절차

1. 아직 안 했다면 레포를 탐색해 코드베이스의 현재 상태를 이해하라. 프로젝트에 컨텍스트 문서(예: `CONTEXT.md`)나 기록된 아키텍처 결정이 있으면 PRD 전반에 그 도메인 어휘를 쓰고, 건드릴 영역의 결정을 존중하라.

2. 기능을 테스트할 seam 을 스케치하라. 새 seam 보다 기존 seam 을 선호하라. 가능한 가장 높은 seam 을 써라. 새 seam 이 필요하면 가능한 가장 높은 지점에 제안하라. 코드베이스 전체의 seam 은 적을수록 좋다 — 이상적인 수는 하나다.

   이 seam 들이 사용자의 기대와 맞는지 확인하라. 사용자가 분명히 PRD 를 한 번에 원하면 막지 마라 — seam 선택을 Implementation Decisions 에 가정으로 적고 사용자가 바로잡게 하라.

3. 아래 템플릿으로 PRD 를 **Markdown** 으로 작성하라. 그다음 **저장을 제안하라** — 특정 도구를 가정하지 마라:
   - 파일로 저장할지(그리고 어디에) 묻고, 그리고/또는
   - **사용자가 이슈 트래커를 설정해 둔 경우에만** 거기에 발행하라(어느 것인지 묻고, 그들이 쓰는 "준비됨" 라벨을 적용하라).

   사용자가 아무 데도 저장을 원치 않으면 Markdown 만 출력하라.

<prd-template>

## Problem Statement

사용자가 겪는 문제를 사용자 관점에서.

## Solution

문제에 대한 해법을 사용자 관점에서.

## User Stories

길고 번호가 매겨진 사용자 스토리 목록. 각 스토리는 다음 형식:

1. <actor> 로서, 나는 <feature> 를 원한다, 그래서 <benefit>.

<user-story-example>
1. 모바일 뱅크 고객으로서, 나는 내 계좌 잔액을 보고 싶다, 그래서 지출에 대해 더 나은 결정을 내릴 수 있다.
</user-story-example>

이 사용자 스토리 목록은 극도로 광범위하며 기능의 모든 측면을 다뤄야 한다.

## Implementation Decisions

내려진 구현 결정 목록. 포함할 수 있는 것:

- 빌드/수정될 모듈
- 수정될 그 모듈들의 인터페이스
- 개발자의 기술적 명료화
- 아키텍처 결정
- 스키마 변경
- API 계약
- 구체적 상호작용

구체적 파일 경로나 코드 스니펫을 넣지 마라. 매우 빨리 낡을 수 있다.

예외: 프로토타입이 산문보다 결정을 더 정밀하게 인코딩하는 스니펫(상태 기계, reducer, 스키마, 타입 모양)을 냈다면, 관련 결정 안에 인라인하고 프로토타입에서 왔다고 짧게 적어라. 작동 데모가 아니라 결정이 풍부한 부분만, 중요한 비트만 다듬어라.

## Testing Decisions

내려진 테스트 결정 목록. 포함:

- 좋은 테스트가 무엇인지에 대한 설명 (구현 세부가 아니라 외부 동작만 테스트)
- 어떤 모듈이 테스트될지
- 테스트의 선행 사례 (즉, 코드베이스의 유사한 종류의 테스트)

## Out of Scope

이 PRD 의 범위 밖인 것들에 대한 설명.

## Further Notes

기능에 대한 추가 노트.

</prd-template>

## 오너 / 리더십 렌즈

CEO/CTO/CFO/PM 를 겸하는 개발자에게 PRD 는 "원하는 것"과 "만들 것" 사이의 계약이다:

- **Problem Statement 는 한 문단으로 된 당신의 전략이다.** 사용자의 문제를 또렷하게 진술하지 못하면 거기에 엔지니어링 돈을 쓸 준비가 안 된 것이다. 그 줄을 먼저 써라; 하류의 모든 것은 구현이다.
- **Out of Scope 는 예산을 지키는 곳이다.** 여기 한 줄 한 줄이 집중을 사는 "아니오"다. 작업 비용을 대는 사람으로서, v1 이 일부러 하지 않는 것을 명시하라.
- **User Stories 는 당신의 수용 기준이자 로드맵이다.** 데모 스크립트와 슬라이스 목록을 겸한다. 데모할 수 없는 스토리는 끝난 것이 아니다.
