# To Issues

수직 슬라이스(트레이서 불릿)를 써서 계획을 독립적으로 집을 수 있는 작업 단위로 쪼갠다.

이 스킬은 계획, 스펙, 또는 PRD — 보통 `to-prd` 스킬의 출력 — 를 소비해 증분으로 슬라이스한다. 필수 이슈 트래커는 없다: 먼저 분해를 Markdown 으로 만들고, 그다음 사용자가 실제로 일하는 곳 어디에든 저장할 것을 제안한다.

## 과정

### 1. 맥락 수집

대화 컨텍스트에 이미 있는 것에서 작업한다. 사용자가 인자로 참조(파일 경로, 이슈 번호, 또는 URL)를 넘기면, 슬라이스하기 전에 가져와 전체를 읽는다.

### 2. 코드베이스 탐색 (선택)

아직 코드베이스를 탐색하지 않았다면, 코드의 현재 상태를 이해하기 위해 한다. 작업 단위의 제목과 설명은 프로젝트의 도메인 용어집 어휘를 써야 하고(`CONTEXT.md` 가 있으면 참고), 건드리는 영역의 ADR 을 존중해야 한다.

구현을 쉽게 만들 prefactor 기회를 찾는다. "변경을 쉽게 만든 다음, 쉬운 변경을 하라."

### 3. 수직 슬라이스 초안

계획을 **트레이서 불릿** 단위로 쪼갠다. 각 단위는 한 레이어의 수평 슬라이스가 아니라, 모든 통합 레이어를 엔드투엔드로 관통하는 얇은 수직 슬라이스다.

<vertical-slice-rules>

- 각 슬라이스는 모든 레이어(스키마, API, UI, 테스트)를 관통하는 좁지만 *완전한* 경로를 전달한다
- 완료된 슬라이스는 그 자체로 데모 가능하거나 검증 가능하다
- 모든 prefactoring 은 먼저 한다

</vertical-slice-rules>

**Prefactor** 단위는 "그 자체로 데모 가능" 규칙의 유일한 허용 예외다 — 사용자 동작을 바꾸지 않고 seam 을 여는 것이라 수용 기준이 "기존 동작 불변, 테스트 통과"다. prefactor 라고 분명히 표시한다; 기능 슬라이스인 척 포장하지 않는다.

stakeholder 가 수평(레이어별) 슬라이스를 요구하면, 근거(끝까지 데모할 게 없음; 통합 리스크가 맨 마지막에 드러남)로 밀어내고 그들의 진짜 목표를 채우는 절충안을 제안한다: **레이어가 아니라 슬라이스 단위로 병렬화하라.** 단일 seam 뒤에 사는 기능은 독립 병렬 작업으로 못 쪼개는 경우가 많다 — 슬라이스들이 점진 체인을 이루며, 각각 그 자체로 데모 가능하되 앞 것 위에 쌓인다. 이는 단일 seam 기능의 본질이지 슬라이싱 실패가 아니다; 억지 독립을 강요하지 말고 그렇게 말하라.

### 4. 사용자에게 퀴즈

제안한 분해를 번호 목록으로 제시한다. 각 슬라이스마다 보여준다:

- **제목**: 짧은 서술적 이름
- **Blocked by**: 어떤 다른 슬라이스가(있다면) 먼저 완료돼야 하는지
- **다루는 사용자 스토리**: 이게 다루는 사용자 스토리(원본 자료에 있으면)

사용자에게 묻는다:

- 입자 크기가 맞는 느낌인가? (너무 굵음 / 너무 잘음)
- 의존 관계가 맞는가?
- 합치거나 더 쪼갤 슬라이스가 있는가?

사용자가 분해를 승인할 때까지 반복한다. 사용자가 분해를 한 번에 원하면(주고받기 없이) 막지 마라 — 입자 크기와 의존성 가정을 명시하고, 슬라이스를 전달하며, 수정을 청한다.

### 5. 출력 및 (선택적) 발행

승인된 슬라이스를 아래 작업 단위 템플릿으로 **Markdown** 으로 렌더한다, 슬라이스당 한 블록, 의존 순서로(blocker 먼저) 정렬해 "Blocked by" 필드가 앞선 슬라이스를 참조할 수 있게.

그다음 저장을 제안한다 — 특정 도구를 가정하지 않는다:

- **GitHub** — `gh issue create` 로 이슈 생성(사용자가 쓰는 "ready" 라벨이 있으면 적용).
- **Linear** — 사용자가 설정한 Linear 연동/CLI 로 이슈 생성.
- **로컬 파일** — 슬라이스당 Markdown 파일 하나(예: `docs/issues/` 아래), 또는 합친 단일 파일.
- **아무것도 안 함** — 그냥 Markdown 을 대화에 남긴다.

사용자가 실제로 트래커를 설정해 둔 경우에만 발행한다; 가정하지 말고 어느 것인지 묻는다. "Blocked by" 에 실제 식별자를 참조할 수 있게 의존 순서로 발행한다. 부모/원본 단위를 닫거나 수정하지 *않는다*.

## What to build

이 수직 슬라이스에 대한 간결한 설명. 레이어별 구현이 아니라 엔드투엔드 동작을 기술한다.

구체적 파일 경로나 코드 스니펫은 피한다 — 빠르게 낡는다. 예외: 프로토타입이 산문보다 더 정확하게 결정을 인코딩하는 스니펫(상태 기계, 리듀서, 스키마, 타입 형태)을 냈다면, 여기 인라인하고 프로토타입에서 왔음을 짧게 적는다. 결정이 풍부한 부분만 다듬는다 — 작동하는 데모가 아니라 중요한 부분만.

## 오너 / 리더십 렌즈

개발자이자 CEO/CTO/CFO/PM 인 사람에게, 슬라이싱이 곧 계획이다:

- **수직 슬라이스는 일찍 데모하고 검증할 수 있는 출시 가능한 증분이다.** 각각은 다음 것을 만들기 전에 무언가를 사용자(또는 자신) 앞에 놓고 배울 기회다. 수평 슬라이스는 맨 끝까지 보여줄 게 없다.
- **의존 순서가 로드맵 시퀀싱이다.** "Blocked by" 그래프는 위장한 당신의 전달 계획이다 — 무엇이 무엇을 푸는지, 임계 경로가 어디를 지나는지 말해준다.
- **첫 슬라이스가 가장 위험한 가정을 증명해야 한다.** 가장 무서운 미지가 먼저 트레이서 불릿을 받도록 순서를 정한다; 막다른 길은 6주 차보다 1주 차에 발견하는 게 낫다.
