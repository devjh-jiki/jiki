## 문서 유형별 템플릿

독자의 할 일에 형태를 맞춘다. 시작점이지 감옥이 아니다 — 안 맞는 섹션은 빼라.

### README (프로젝트 정문)

독자는 "이게 나한테 맞나, 어떻게 시작하지"를 결정하는 중. 빨리 돌아가게 해준다.

```markdown
# 프로젝트 이름
한 문장: 무엇이고 누구를 위한 것인가.

## 왜 / 무엇을 하는가
가치를 2~4문장 또는 짧은 불릿으로.

## 빠른 시작
작동하는 결과까지 복붙 가능한 단계. 번호 매기고, 최소한으로.

## 사용법
사람들이 가장 많이 하는 2~3가지, 예시와 함께.

## 설정 / 옵션
필요할 때만. 옵션 표, 기본값, 각각의 역할.

## 링크
문서, 기여 가이드, 라이선스 — 더 깊은 자료로 바깥을 가리킨다.
```

### How-to 가이드 (작업 지향)

독자에게 구체적 목표가 있다. 시작부터 완료까지 순서대로 데려간다.

```markdown
# <작업>을 하는 법
끝에 무엇을 갖게 되는지(한 문장) + 사전 조건.

## 단계
1. 행동. 예상 결과.
2. 행동. 예상 결과.
...

## 잘 됐는지 확인
독자가 성공을 확인하는 법.

## 문제 해결
흔한 실패 → 원인 → 해결.
```

### 레퍼런스 (조회 지향)

독자는 원하는 걸 알고 정확한 디테일이 필요하다. 서사가 아니라 훑기와 완전성에 최적화.

```markdown
# <대상> 레퍼런스
한 줄 범위.

## <항목> (예: 함수 / 엔드포인트 / 플래그)
- **시그니처 / 문법**
- **매개변수** — 표: 이름, 타입, 필수 여부, 설명
- **반환 / 응답**
- **예시** — 복붙 가능
- **참고 / 에러**
```

### 결정 기록 (ADR)

되돌리기 어려운 결정과 그 트레이드오프를 담아, 미래 독자가 *왜*를 알게 한다.

```markdown
# ADR-NNN: <결정 제목>
**상태:** 제안 | 수용 | 대체됨
**날짜:** YYYY-MM-DD

## 맥락
작용한 힘들 — 무엇이 이 결정을 필요하게 만들었나.

## 결정
무엇을 골랐는지, 분명하게.

## 결과
이게 쉽게 만드는 것, 어렵게 만드는 것, 포기하는 것.

## 검토한 대안
다른 선택지들과 왜 졌는지.
```

### 리서치 브리프 (조사 결과)

조사 + 검증을 신뢰할 수 있고 출처 있는 글로 바꾼다.

```markdown
# <주제> — 리서치 브리프
**기준일:** YYYY-MM-DD · **신뢰도:** 높음/중간/혼재

## 요약
중요한 발견 3~5개, 각 한 줄. 답을 앞세운다.

## 발견
### <주장 1>
주장, 근거, 출처(인라인 링크). 상하기 쉬우면 신뢰도와 날짜 표기.

## 열린 질문 / 공백
확인 못 한 것, 확인하려면 무엇이 필요한지.

## 출처
백업 목록 — 1차 출처부터, 발행처와 날짜와 함께.
```

## 연결 패턴

좋은 문서 묶음은 파일 더미가 아니라 탐색 가능한 그물이다.

- **필요한 지점에 링크한다.** 다른 곳에서 다룬 개념을 언급할 땐 바로 그 자리에 인라인 링크 — 독자가 찾아 헤매게 하지 마라.
- **관련 문서는 양방향으로 링크.** A가 더 깊은 B를 링크하면, B도 맥락으로서 A를 역링크해야 한다. 단방향 링크는 막다른 길을 만든다.
- **레포 안에서는 상대 링크** (`./guide.md`, `../adr/0003.md`)를 써서 클론해도 살아남고 포크에서 깨지지 않게. 절대 URL은 진짜 외부 출처에만.
- **모든 문서에 "다음" 또는 "관련" 포인터를.** 한 문서를 끝낸 독자가 어디로 갈지 알아야 한다. 맨 아래 짧은 "See also"·"다음 단계"가 이 역할.
- **긴 문서엔 앵커 링크.** 위쪽 목차 + `#제목` 앵커로 점프 가능하게. GitHub은 제목에서 앵커를 자동 생성한다(소문자, 공백→하이픈).
- **과하게 링크하지 마라.** 한 섹션에서 같은 대상을 다섯 번 링크하는 건 소음. 첫 언급에만.

## 개발자 문서 장인 정신

최고의 개발자 문서(Stripe, Twilio, 클라우드 제공사 문서의 좋은 부분)가 공유하는 패턴. 플랫폼 비종속이다 — 특수 문법 없이 순수 GitHub 마크다운에서 동작한다. 문서에 코드·통합·API가 있으면 언제든 적용하라.

- **명사가 아니라 할 일로 제목을 단다.** "결제 페이지 만들기"가 "Checkout API"보다, "개발환경 세팅하기"가 "환경"보다 낫다. 독자는 할 일을 갖고 도착한다 — 제목에 그 일을 적어 제대로 찾아왔음을 알게 하라. 명사 제목은 '실행'이 아니라 '조회'가 목적인 순수 레퍼런스 페이지에만.
- **코드 블록은 완전하고 복붙 가능해야 한다.** 실행되는 스니펫이 중간에 `...`이 든 조각보다 낫다. 독자가 붙여넣어 바로 동작하면(또는 뻔한 빈칸 하나만 채우면 동작하면) 디버깅 한 세션을 아껴준 것이다. 흥미로운 한 줄만이 아니라 import와 셋업도 보여라.
- **'무엇'이 아니라 '왜'를 주석으로.** `// 클라이언트가 금액을 조작하지 못하도록 서버에서 합계 계산`은 가르치고, `// 합계 계산`은 소음이다. 인라인 주석은 코드만 봐선 알 수 없는 이유를 설명하는 자리다 — 비자명한 것과 보안 관련에 써라.
- **예시에 진짜 시크릿을 넣지 말고, 경고하라.** 분명한 플레이스홀더(`<<YOUR_API_KEY>>`, `sk_test_...`)를 쓰고, 키는 커밋된 코드에 들어가면 안 된다는 한 줄 경고를 단다. 안전하지 않은 패턴을 보여주는 문서는 그걸 가르친다.
- **방법이 여럿이면 하나를 권하고 이유를 말한다.** "대부분의 통합엔 Payment Intents보다 Checkout Sessions를 써라, 코드가 훨씬 적으니까"가 둘을 중립적으로 나열하는 것보다 가치 있다. 권장 경로를 앞세우고, 대안과 바꿀 구체적 조건을 언급하라. 독자가 네 의견을 역추적하게 만들지 마라.
- **안전하게 시험하는 법을 알려준다.** 테스트/샌드박스/드라이런 경로, 샘플 데이터, "이건 실제 청구 안 됨" 같은 메모가 따라하기 비용을 낮춘다. 1단계가 "실물을 건다"가 아니면 독자가 실제로 단계를 돌려볼 확률이 높아진다.
- **깊이별로 층을 나누고 층끼리 링크한다.** 개발자 문서는 사다리로 작동한다: 몇 분 안에 *뭔가* 돌아가는 quickstart, 그다음 더 충실한 가이드, 그다음 빠짐없는 레퍼런스. 각 칸을 한 깊이에 두고 위/아래로 링크하라, 셋을 한 페이지에 욱여넣지 말고. 완전해지려는 quickstart는 더 이상 quick하지 않다.

## 가독성 체크리스트

**다듬기** 때 대상 독자가 되어 돌린다.

- [ ] 문서 맨 위가 낯선 사람을 ~10초 안에 방향 잡아준다: 이게 뭔지, 나한테 맞는지, 어디서 시작하는지.
- [ ] 각 섹션이 결론으로 시작하고 뒤에서 뒷받침한다.
- [ ] 한 문단 한 아이디어. 문장이 첫 읽기에 파싱될 만큼 짧다.
- [ ] 구조가 내용에 맞다: 평행 항목엔 목록, 비교엔 표, 추론엔 산문. 잘못된 형태에 욱여넣은 것 없음.
- [ ] 출처에 기대는 모든 주장에 출처가 인라인으로 링크됨.
- [ ] 추상적 설명이 아니라 구체적 예시·명령·표가 무게를 진다.
- [ ] 군더더기 없음: 목 가다듬는 서론, "이 섹션에서는 ~할 것이다", 중복 재진술 — 잘라냄.
- [ ] 제목이 문법적으로 평행하고, 그 자체로 개요처럼 훑힌다.
- [ ] 전문용어는 첫 사용 시 정의하거나, 정의된 곳으로 링크.

## 마크다운 메커닉 체크리스트

조용히 렌더링이나 신뢰를 깨뜨리는 것들.

- [ ] 코드 펜스에 언어 태그(```` ```python ````, ```` ```bash ````)가 있어 하이라이팅됨.
- [ ] 모든 코드 블록이 복붙 가능하고 실제로 정확함 — 가능하면 명령을 테스트.
- [ ] 표가 GFM 파이프 문법을 쓰고 렌더링됨(헤더 행 + 구분 행 존재).
- [ ] 내부 링크가 풀림(상대 경로 정확); 외부 링크가 살아있음, 404 아님.
- [ ] 제목 레벨이 일관되고 건너뛰지 않음(`#`에서 `###`로 점프 없음).
- [ ] 이미지에 alt 텍스트 있음; 경로 풀림.
- [ ] 그 플랫폼이 대상이 아니면 플랫폼 전용 문법 없음(Obsidian `[[wikilink]]`, Docusaurus `:::note` 등).
- [ ] 긴 문서(~2화면 초과)에 목차 있음.
- [ ] 라인 단위: 끝 공백 잔여물 없음, 일관된 목록 마커, 코드 블록·표 주위 빈 줄로 렌더링 보장.
