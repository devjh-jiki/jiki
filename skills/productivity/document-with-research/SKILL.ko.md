# document-with-research

주제 하나, 흩어진 노트 더미, 혹은 거친 기존 문서를 명확하고 신뢰할 수 있으며 잘 연결된 마크다운으로 바꾼다. 이 스킬은 **조사 → 검증 → 설계 → 작성 → 다듬기** 전체 파이프라인을 다루지만, 다섯 단계가 다 필요한 경우는 드물다. 사용자가 실제로 어디에 있는지 진단하고 거기서 시작하라.

## 먼저: 진입 지점을 찾는다

반사적으로 전체 파이프라인을 돌리지 마라. 요청과 손에 있는 자료를 읽고, 사용자가 원하는 걸 주는 가장 가벼운 경로를 고른다.

- **"X 조사해서 문서로"** → **조사**부터 시작, 전체 파이프라인 실행.
- **"내 사이드 프로젝트 문서화 / README·가이드 작성"** → 사실은 이미 코드와 사용자 머릿속에 있다. 외부 출처가 필요한 주장이 아니면 웹 조사는 건너뛰고 **설계**부터 시작.
- **"이 문서 정리/개선/재구성"** → **다듬기**부터 시작. 처음부터 다시 쓰지 말고, 있는 걸 존중하면서 약한 곳을 고친다.
- **"이거 팩트체크 / 출처 달아줘"** → 기존 텍스트에 **검증**을 돌리고, 가벼운 **다듬기**로 인용을 녹여 넣는다.

어느 경로일지 모르겠으면 추측하지 말고 날카로운 질문 하나를 던진다 — 단, 요청이 명확하면 그냥 진행한다. 아래 파이프라인은 풀 버전이다. 상황에 맞게 줄여라.

**독자의 언어로 쓴다.** 요청과 기존 자료의 언어에 맞춘다 — 사용자가 한국어로 말을 걸거나 한국어 레포를 건네면 문서는 한국어, 영어 요청이면 영어. 템플릿이 마침 영어라는 이유로 조용히 언어를 바꾸지 마라. 문서는 템플릿이 아니라 독자를 위한 것이다.

## 파이프라인

### 1. 조사 (Research)

문서가 아직 없는 사실에 의존할 때, 쓰기 전에 모은다.

- **의도를 갖고 검색한다.** 각 검색은 적어둔 구체적 질문 하나에 답해야 하지, "X에 대해 알아보기"가 아니다. 열린 질문 목록을 짧게 유지하며 하나씩 닫는다.
- **1차·권위 있는 출처를 우선한다.** 공식 문서, 표준 기관, 정부·오픈데이터 포털, 프로젝트 소스코드, 동료심사 자료 — 블로그·집계 사이트보다 우선. 2차 출처를 인용할 땐 1차까지 거슬러 올라가 보라.
- **출처는 그때그때 기록한다**, 나중이 아니라: URL, 누가 발행했는지, 언제, 그게 뒷받침하는 구체적 주장. 인용과 검증에 필요하다.
- **중요한 열린 질문이 닫히면 멈춘다.** 수확 체감은 실재한다. 영원히 조사하지 마라.

소스 티어링, 오픈데이터 포털, 출처 충돌 처리는 `references/research-and-verify.md` 를 읽어라.

### 2. 검증 (Verify)

조사는 주장을 찾고, 검증은 그중 발행할 만큼 믿는 것을 가른다.

- **사실과 추론을 분리한다.** 공식 출처의 숫자는 사실. "이건 시장이 커지고 있음을 시사한다"는 네 추론이다 — 그렇게 표시하고, 사실인 척 끼워 넣지 마라.
- **하중을 받는 주장은 교차 확인한다.** 독자가 행동의 근거로 삼을 것, 틀리면 문서가 망신당할 것은 독립된 2차 출처를 받을 자격이 있다.
- **상하기 쉬운 건 전부 날짜를 단다.** 가격, 버전, "현재" 1위, 통계 — as-of 날짜를 적어 독자가 유통기한을 알게 한다.
- **공백에 정직하되, 공백을 먼저 벌어라.** 정직함은 *진짜 시도한 뒤*의 fallback이지, 시도를 건너뛰는 지름길이 아니다. "미확인"이라고 쓰기 전에 뻔한 다음 수를 해보라: 포털 페이지가 막혔으면 데이터 API·내보내기를 시도, 검색엔진이 실패하면 출처 도메인으로 직접, 쿼리 하나가 비면 바꿔서 다시. 진짜 시도가 실패했을 때만 확실한 것으로 반올림하지 말고 "미확인" / "X 시점 기준, Y 출처"로 표시한다. 조금만 더 파면 나올 숫자를 항복하지 마라.

### 3. 설계 (Design)

산문을 쓰기 전에 형태를 정한다. 문서의 구조가 품질의 대부분이다.

- **독자와 단 하나의 할 일을 명명한다.** 누가 이걸 열고, 떠날 때 무엇을 할 수 있어야 하나? 시작 가이드, API 레퍼런스, 결정 기록(ADR)은 형태가 다르다 — 할 일에 형태를 맞춘다.
- **개요(outline)부터.** 문단 하나 쓰기 전에 제목 계층을 짜고, 그게 독자를 "내가 왜 여기 있지"에서 "이제 행동할 수 있다"까지 데려가는지 확인한다. 개요는 재배치가 싸고, 산문은 비싸다.
- **무엇이 무엇에 연결되는지 정한다.** 문서는 좀처럼 홀로 서지 않는다. 이게 어떤 기존 문서에 링크해야 하고 어디서 역링크받아야 하는지 적어, 섬이 아니라 탐색 가능한 그물이 되게 한다.

문서 유형별 템플릿(README, 가이드, 레퍼런스, ADR, 리서치 브리프)과 연결 패턴은 `references/markdown-quality.md` 를 읽어라. 문서가 개발자 대상이면 — 통합 가이드, API/CLI 문서, 코드가 들어가는 무엇이든 — 그 파일의 "개발자 문서 장인 정신" 섹션도 적용하라: 할 일로 제목 달기, *왜*를 주석으로 단 완전한 복붙 코드, 여러 방법 중 하나를 권하고 이유 말하기, quickstart → 가이드 → 레퍼런스로 층 나누기.

### 4. 작성 (Write)

개요를 채운다. 목표는 독자가 요점을 빨리 잡고 신뢰하는 것.

- **답을 앞세운다.** 각 섹션을 결론으로 시작하고 뒤에서 뒷받침한다. 독자는 훑는다. 훑기에 보상하라.
- **한 문단 한 아이디어, 짧은 문장.** 구조가 도움 될 땐 빽빽한 산문을 목록·표로 쪼개되, 사실은 산문인 것을 억지로 목록화하지 마라.
- **보여주고 나서 말한다.** 구체적 예시, 명령 블록, 표는 추상적 설명보다 세게 꽂힌다. 모든 코드 블록은 복붙 가능하고 정확해야 한다.
- **주장이 출처에 기대면 인라인으로 인용한다.** 맨 아래 참고문헌 더미가 아니라 주장하는 그 지점에 출처를 링크하라.

### 5. 다듬기 (Refine)

내가 썼든 물려받았든, 비판적 독자로서 한 번 훑는다.

- **제 몫 못 하는 건 잘라낸다.** 중복 문장, 목 가다듬는 서론, "이 섹션에서는 ~할 것이다". 조여라.
- **흐름을 고친다.** 각 섹션이 앞 섹션에서 따라 나오나? 제목이 평행하고 훑기 좋은가? 문서 맨 위가 낯선 사람을 10초 안에 방향 잡아주나?
- **메커닉을 점검한다.** 작동하는 링크(내부·외부), 렌더링되는 표, 언어 태그 붙은 올바른 코드 펜스, 일관된 제목 레벨, 이미지 alt 텍스트.
- **대상 독자로서 다시 읽는다.** 코딩 시작한 배관공과 스태프 엔지니어는 필요한 게 다르다. 실제 독자가 따라올 수 있는지 확인하라.

끝났다고 부르기 전에, `references/markdown-quality.md` 의 가독성·메커닉 체크리스트를 기억이 아니라 명시적 최종 패스로 돌려라. 중복된 표 행, 재작성 중 조용히 사라진 사실, 깨진 링크, 태그 없는 코드 펜스가 여기서 잡힌다. 최종 패스를 건너뛰는 게 작은 결함이 발행본까지 살아남는 경로다.

가독성·마크다운 메커닉 전체 체크리스트는 `references/markdown-quality.md` 를 읽어라.

## 산출 규칙

- **이식 가능한 마크다운을 목표로** 한다. 기본은 GitHub(GFM)에서 올바르게 렌더링: 언어 태그 붙인 펜스 코드, GFM 표, 같은 레포 문서 간 상대 링크. 사용자가 그 대상이라고 말하지 않는 한 플랫폼 전용 문법(Docusaurus admonition, Obsidian `[[wikilink]]`)은 피한다.
- **문서를 탐색 가능하게.** 두어 화면을 넘는 건 위쪽에 짧은 목차를 단다. 관련 문서를 양방향으로 링크해 그물을 이룬다.
- **출처를 보이게 유지한다.** 주장 지점에 인라인 인용. 맨 아래 "Sources"·"References" 섹션은 백업으로 좋지만, 인라인 링크의 대체물은 아니다.
- **기존 레포 컨벤션에 맞춘다.** 프로젝트에 이미 문서 스타일(제목 깊이, `docs/` 파일 배치, 네이밍)이 있으면 새 걸 강요하지 말고 따른다.

## 왜 중요한가

문서는 쓰이는 것보다 훨씬 자주 읽힌다. 보통 당신이 아닌, 물어볼 수도 없는 누군가에게. 잘 조사됐지만 구조 없는 문서는 독자의 시간을 낭비하고, 아름답게 구조화됐지만 틀린 문서는 독자가 오류를 잡는 순간 신뢰를 무너뜨린다. 파이프라인은 둘 다 지키려 존재한다 — 검증으로 사실이 버티게, 설계·다듬기로 독자가 마찰 없이 그 사실에 닿게. 단계를 건너뛸 땐 — 자주 그래야 한다 — 이 문서에 필요 없는 단계를 건너뛰어라, 그저 귀찮은 단계가 아니라.
